# 可记忆自进化 Agent 底座：架构详细设计报告（v0.2）

## 1. 设计目标

本设计严格服务“底层能力优先”原则：先解决记忆、进化、意图、安全、执行总线，再逐步完善上层体验。

核心目标：

- 支撑“长期可托付”：跨会话、跨设备、跨场景连续工作。
- 支撑“可解释可治理”：每次关键决策可回溯、可审计、可回滚。
- 支撑“越用越好”：经验自动沉淀为技能，技能可验证迭代。
- 支撑“具身接入”：与用户共享环境感知（视觉/听觉/空间实体线索）。

---

## 2. 总体架构（逻辑分层）

### 2.1 功能分层与横切能力

在六层功能架构之外，显式引入 **集成中枢层（Integration Hub）**：作为 `Hermes`、`MemVerse`、`Leaper`、`Holomime`、`Reachy` 等异构组件之间的 **唯一推荐通信通道**，避免组件间直接点对点调用演变成「意大利面条式集成」。**安全与主权治理层** 覆盖全部层级，对澄清策略与高风险动作实施硬约束。

首期实施宜 **薄总线 + 强约定的事件/TaskSpec 契约 + 按组件 Adapter**，版本升级的影响应限制在对应适配器内，而非扩散到业务编排逻辑。

```
┌─────────────────────────────────────┐
│     安全与主权治理层（全覆盖）         │
│        Holomime 思路 + 自建           │
└─────────────────────────────────────┘
┌─────────────────────────────────────┐
│        集成中枢层（Integration Hub）   │
│   统一消息路由、格式转换、版本缓冲     │
└─────────────────────────────────────┘
┌──────────┐ ┌──────────┐ ┌──────────┐
│ 感知接入层 │ │意图建模层 │ │ 执行编排层 │
│ Reachy…  │ │+澄清决策树│ │+MCP 工具总线│
└──────────┘ └──────────┘ └──────────┘
┌──────────────────┐ ┌──────────────────┐
│  记忆与知识层      │ │ 自我进化层         │
│ MemVerse+Wiki    │ │ Leaper 机制        │
│ +查询沉淀+生命周期  │ │ +负向验证+多样性   │
└──────────────────┘ └──────────────────┘
```

采用六层功能架构：

1. **感知接入层（Perception I/O）**
2. **意图与任务建模层（Intent Modeling）**
3. **执行编排层（Execution Orchestration）**
4. **记忆与知识层（Memory Fabric）**
5. **自我进化层（Evolution Engine）**
6. **安全与主权治理层（Governance Plane）**

运行时主干建议：
- `Hermes/OpenClaw` 承担会话与工具调度（工具侧优先经 MCP Client 接入，见 3.3）。
- `MemVerse` 承担长期记忆中台。
- `Leaper` 机制承担进化引擎。
- `Holomime` 思路承担治理硬约束。
- `Reachy Mini` 作为具身终端。

**集成中枢层职责（摘要）**
- **数据格式转换**：各组件内部表示 → 统一事件 / 指令模型。
- **协议适配**：例如 Hermes 工具协议与 OpenClaw 插件体系的对外统一封装。
- **版本缓冲**：依赖组件升级时，在 Adapter 内做兼容，业务层保持稳定契约。

**验收要点**：任意单一依赖组件升级时，对其余组件的影响应可收敛在对应 Adapter 内。

## 3. 关键模块详细设计

## 3.1 感知接入层（Perception I/O）

**用户需求表达**  
“Agent 要与我共享现场，而不是等我每次解释背景。”

**输入通道**
- 语音流（麦克风/设备）
- 视频帧（摄像头）
- 环境状态（时间、位置、日程、设备状态）
- 交互事件（聊天消息、任务反馈、操作结果）

**设计要点**
- 使用 `Reachy Mini` daemon + WebRTC/IPC 架构，确保本地低延迟与远程一致能力。
- 感知数据事件化，统一写入 `observe_event` 流，不直接耦合上层模型。
- 多模态输入先做轻量预处理（ASR、人脸/说话人识别），再入意图层与记忆层。

**核心数据结构（示意）**
```json
{
  "event_id": "obs_...",
  "ts": "2026-05-01T15:00:00Z",
  "modality": ["audio", "vision"],
  "entities": [{"id":"person_x","type":"person","confidence":0.91}],
  "scene": {"location":"office_meeting_room","activity":"meeting_end"},
  "raw_refs": {"audio":"s3://...","frames":"s3://..."}
}
```

## 3.2 意图与任务建模层（Intent Modeling）

**用户需求表达**  
“我说得不完整，它也能理解真正目标，并在关键风险点问我确认。”

**职责**
- 从自然语言 + 场景上下文抽取目标、约束、风险、完成标准。
- 输出结构化 `TaskSpec` 作为执行编排层输入。
- 维护澄清策略：高风险强确认，低风险少打扰。
- **澄清决策树（硬约束，由治理层强制执行）**：LLM 仅负责将意图映射到操作分级；具体是否确认、如何确认 **不由模型覆盖**。分级定义如下（可按产品形态微调 L3 的强认证方式，如无硬件则采用口令/二次验证等「强认证通道」）。

| 分级 | 示例 | 澄清策略 |
|------|------|----------|
| L0 查询 | 查天气、查日历、问知识库 | 零确认，直接执行 |
| L1 内部写入 | 创建笔记、更新 Wiki、设置提醒 | 执行后通知，无需事前确认 |
| L2 外部通信 | 发邮件、发消息、创建会议邀请 | 草稿预览，一键确认 |
| L3 财务/权限 | 购买、转账、修改系统设置 | 强制二次确认 + 强认证通道（含可选生物识别） |
| L4 他人关系 | 代表用户向他人承诺、拒绝邀请 | 暂停执行，等待显式授权 |

**TaskSpec（建议）**
```json
{
  "task_id": "task_...",
  "goal": "安排下周项目风险复盘并输出会议纪要",
  "constraints": ["下周二前完成", "仅团队核心成员"],
  "preferences": ["简洁风格", "结论先行"],
  "risk_level": "medium",
  "confirm_policy": "key_step_only",
  "success_criteria": ["会议已创建", "纪要已归档并通知成员"]
}
```

**技术实现建议**
- 运行时主模型做初步解析。
- 规则层做风险分级与字段校验（避免“看似结构化，实则空字段”）。
- 关键任务对象写入记忆层（可追溯）。

## 3.3 执行编排层（Execution Orchestration）

**用户需求表达**  
“我只交代结果，不想管步骤；失败时它要有替代路径。”

**职责**
- 将 `TaskSpec` 转为可执行 DAG（步骤、依赖、重试、补偿）。
- 统一调用工具（文件、邮件、日历、浏览器、设备控制等）。
- 对执行过程进行状态追踪、错误分类与回滚。

**设计要点**
- 采用「统一执行总线」：**优先以 MCP（Model Context Protocol）作为工具接入协议**。Agent 侧实现 MCP Client（工具发现、调用、结果解析）；高频通用能力尽量复用社区 MCP Server（文件、数据库、浏览器、日历等）；**专有接口**（如 Reachy Mini 控制、`MemVerse` 记忆 API、尚未进入生态的内部服务）通过 **自建 MCP Server 适配器** 暴露，避免再回到分散的自定义插件注册。
- 传输：按需支持 stdio（本地子进程）与 SSE（远程服务），在安全与扩展性之间权衡。
- 标准返回：`success`、`retryable`、`side_effect`、`audit_ref`。
- 失败分级：可自动重试 / 需人工确认 / 终止并告警。

**执行事件（示意）**
```json
{
  "task_id": "task_...",
  "step_id": "step_send_calendar_invite",
  "tool": "calendar.create_event",
  "input_hash": "sha256:...",
  "result": {"ok": true, "event_id": "cal_123"},
  "latency_ms": 842,
  "retry_count": 0
}
```

## 3.4 记忆与知识层（Memory Fabric）

**用户需求表达**  
“它要记住我们共同经历，不是只记聊天文本。”

**三层记忆模型**
- **短期记忆（STM）**：当前任务上下文，秒级/分钟级。
- **长期记忆（LTM）**：实体、事件、偏好、关系，图结构组织。
- **参数记忆（PM）**：从长期记忆蒸馏出的高频知识，提高响应速度。

**写入策略**
- 只写“高价值记忆”：决策、偏好变化、关键事件、高复用流程。
- 写入前过门控：去重、冲突检测、可信度评分、敏感信息策略检查。
- 支持时间衰减与晋升：工作记忆可晋升为核心记忆。

**建议实现**
- `MemVerse` 作为独立 memory service（API/MCP）。
- 运行时只通过 `memory_service` 访问，避免业务代码直连底层存储。
- 可并行保留 Markdown/Wiki 镜像层（LLM Wiki），增强可读性与可审计性。

**查询沉淀（Query Archival）闭环**  
区别于传统 RAG 的单次检索，显式支持「好答案归档为 Wiki 页面」，使探索在知识库中复利：
- **触发提示**：当一次问答触发多页综合推理、或形成新的跨实体关联时，可提示沉淀。
- **沉淀内容**：查询结论、推理路径、引用源页面整理为新 Wiki 页；更新相关实体页与 `index` 链接。
- **用户确认**：沉淀前预览，可选「归档 / 仅回复 / 丢弃」。
- **验收建议**：连续使用一周后，`index.md` 中出现若干带明确来源标注（摄入 vs 查询沉淀）的新条目。

**记忆生命周期与陈旧化**
- 在门控与检索侧引入 **`lifespan_score`**（或等价字段）：综合时效、重要性、访问频率，检索时加权而非简单等权召回。
- 区分「短期不活跃但仍重要」与「应降级/遗忘」；用户表达「不重要」时，联动记忆与派生 Skill 降权。
- 周期性记忆 Lint：除矛盾检测外，对长期未访问且可能过期的条目触发相关性确认（如「关于 X 项目的记忆已 6 个月未用，是否仍有效？」）。

## 3.5 自我进化层（Evolution Engine）

**用户需求表达**  
“我纠正一次，它下次就会；学会后还能持续优化。”

**六段进化闭环（借鉴 Leaper）**
- `L0` 混合召回：BM25 + 向量。
- `L1` 经验提炼：从任务回放提取结构化经验。
- `L2` 技能生成：经验聚类并合成可复用 Skill。
- `L3` 技能演化：合并、淘汰、晋升、漂移检测。
- `L4` 用户建模：更新个人偏好与决策风格。
- `L5` 验证守护：一致性检查、回归测试、时间衰减。

**Skill 对象建议**
```json
{
  "skill_id": "skill_...",
  "name": "会议纪要自动归档与通知",
  "trigger_conditions": ["meeting_end", "notes_generated"],
  "procedure": ["format_summary", "archive_to_wiki", "notify_channel"],
  "guardrails": ["不得包含敏感字段", "外发前二次检查"],
  "confidence": 0.72,
  "version": 3,
  "status": "active"
}
```

**关键原则**
- Skill 上线必须灰度（shadow/limited rollout）。
- 任何自动演化必须可回滚（版本化 + 生效窗口）。
- 进化收益要量化（成功率、时延、用户修正率）。

**负向验证（防「错误学习」放大）**  
从成败提炼 Skill 时，表面「成功」可能来自偶然场景。每个新 Skill 宜同步生成 **1～3 个反例用例**（明确不应触发该 Skill 的情境），在灰度期检测误触发率；超过阈值则回炉，避免错误 Skill 因「简单场景反复成功」而虚高置信度。

**多样性保持（防单用户过拟合）**  
- 高频行为模式不立即固化为永久唯一偏好；可保留 Skill 多版本/多风格变体并定期轮替评测。
- 当短期指令与用户自述长期目标（健康、财务等）冲突时，允许「善意提醒」类策略（仍由治理规则约束，而非模型随意发挥）。

## 3.6 安全与主权治理层（Governance Plane）

**用户需求表达**  
“它能力越强，越要可控，不允许越权替我做主。”

**治理策略**
- 数据分级：公开/内部/敏感/极敏感。
- 操作分级：查询、编辑、外发、财务、账号操作等。
- 授权机制：低风险自动，高风险二次确认，极高风险强阻断。
- 审计链：决策依据、调用参数、执行结果全链路记录。

**硬约束执行**
- 借鉴 Holomime `conscience` 思路，关键规则编译执行，不依赖 LLM 意图“自觉”。
- 对高风险动作启用 deny/allow/escalate 三态判定。
- 一键紧急刹车（kill-switch）覆盖执行层。

## 4. 端到端数据流（从一句话到长期学习）

1. 用户发起请求（文本/语音） -> 感知层生成 `observe_event`，经集成中枢规范化后分发。  
2. 意图层生成 `TaskSpec`、映射操作分级（L0–L4）；治理层按澄清决策树强制执行确认策略。  
3. 执行层规划 DAG，经 **MCP 工具总线**（及专有 MCP Server）调用工具。  
4. 过程与结果写入审计日志与短期记忆。  
5. 记忆层门控（含 `lifespan_score`）后入长期记忆；可选查询沉淀进入 Wiki。  
6. 进化层按周期提炼经验并产出/更新 Skill（含反例验证与灰度）。  
7. 治理层持续评估策略命中与异常行为，必要时阻断。

## 5. 技术栈建议（与开源映射）

- **运行时与工具调度**：`Hermes`（优先）或 `OpenClaw`
- **工具接入协议**：**MCP**（Client + 社区/自建 Server）；通用工具优先复用生态，专有 API 自建 MCP Server
- **组件间集成**：集成中枢层（事件模型 + Adapter），可选轻量消息路由实现
- **记忆中台**：`MemVerse`
- **进化引擎**：`Leaper` 机制抽取实现
- **治理与人格边界**：`Holomime` 规则与身份栈
- **具身端**：`Reachy Mini SDK`
- **多模态记忆图参考**：`M3-Agent`

## 6. API 与组件边界（建议先定）

建议先定义内部接口，降低后续耦合（可与 MCP 暴露面并存：内部契约稳定，对外工具走 MCP）：

- `integration_hub.emit/route()`：统一事件入站、出站与 Adapter 挂载点（命名可按实现调整）
- `perception_service.observe()`
- `intent_service.parse(observe_event, history_context)`
- `executor_service.run(task_spec)`（内部调用 MCP Client）
- `memory_service.write/read/search()`
- `evolution_service.reflect_and_update(task_trace)`

每个接口都返回标准 `trace_id`，统一接入审计与观测系统。

## 7. 非功能性要求（底座阶段必须满足）

- **可靠性**：任务执行可重试、可恢复、可补偿。
- **可观测性**：日志/指标/追踪三件套齐全。
- **可解释性**：关键决策必须给出依据链路。
- **可运维性**：支持单机部署与云端部署两套模式。
- **可扩展性**：新工具、新记忆后端、新模型可插拔。

## 8. 阶段性交付建议（仅底座）

### 阶段 A：可追溯记忆底座 + 集成中枢骨架
- 打通 observe -> memory -> recall。
- 完成记忆门控与审计日志。
- **同步落地集成中枢层最小骨架**（事件/schema、至少一条端到端 Adapter 路径），避免后期大规模重构。
- 验收：关键事实可回溯、误记忆可纠正；**单一依赖组件 mock 升级时，业务编排代码无感知或仅改 Adapter**。

### 阶段 B：意图到执行闭环（澄清规则 + MCP）
- 打通 TaskSpec -> DAG -> **MCP 工具总线** -> result。
- **落地澄清决策树（L0–L4）**，与治理层联动。
- 完成失败重试与分级确认点。
- 验收：高频任务可稳定自动完成；高风险路径 **100%** 按分级策略进入确认。

### 阶段 C：进化闭环 + 负向验证
- 打通经验提炼 -> Skill 生成 -> **反例灰度** -> 回归验证。
- 验收：连续两周可观察到可量化改进；新 Skill 均有反例抽检记录。

### 阶段 D：治理闭环 + 记忆生命周期
- 完成分级授权、硬约束规则、紧急刹车。
- **落地 `lifespan_score`、主动遗忘与周期性相关性 Lint**；Wiki 查询沉淀可选与本阶段并行成熟。
- 验收：高风险动作 100% 进入治理链路；检索能体现时效加权与用户「不重要」反馈。

---

## 9. 成功标准（底座是否可用）

满足以下四条，才算“底座可用”：

- 记忆可用：能记住并正确召回关键长期事实。
- 进化可用：新经验能转化为可验证可复用技能。
- 治理可用：高风险动作受控且可审计。
- 执行可用：任务闭环稳定，失败可恢复。

---

## 10. 本版结论

从工程现实看，你的最佳路径不是先做“全能体验”，而是先形成一个“可治理的自进化底座”。  
v0.2 相对 v0.1 的核心增补：**集成中枢层**（控制多组件胶水复杂度）、**澄清决策树**（确认策略可规则化）、**MCP 优先的工具总线**、**LLM Wiki 查询沉淀**、**记忆生命周期**、进化侧 **负向验证与多样性保持**。  
该底座一旦建立，感知、人格、社交、跨场景能力都能在同一骨架上持续生长，而不是反复推倒重来。
